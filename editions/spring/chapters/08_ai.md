## Think about it

**1.** An AI generates a clue. The creator approves it without reading carefully. A player spends 40 minutes on that clue because the AI's phrasing was subtly ambiguous. Who is responsible for the bad clue?

**2.** "AI-assisted" and "AI-generated" are different. What changes when the human goes from editor to approver?

**3.** A difficulty estimator predicts that a clue will have a 70% first-attempt success rate. The actual rate turns out to be 35%. Is the estimator wrong, or was it trained on a different population?

---

## Architecture of the sidecar

The Python AI sidecar is a FastAPI service running at `http://localhost:8001`. The Rust API talks to it via HTTP. The separation is deliberate:

**Language boundary.** The Python ecosystem has richer ML tooling — LLM clients, vision libraries, and fast experimentation loops. Rust has better performance and concurrency. The sidecar does the ML; the Rust API does everything else.

**Failure isolation.** If the AI sidecar goes down, the non-AI parts of the game still work: text, QR/NFC, and GPS answers can be submitted; static hints can be requested; hunts can be completed. The degraded functions are AI clue drafting, difficulty estimation, semantic fallback matching, photo verification, adaptive hints, and session analysis. This is the "remove the AI and it still works, just less well" property.

**Independent scaling.** The sidecar's work is latency-bound on external LLM API calls — and GPU-bound only if you self-host a model via Ollama. The Rust API is I/O-bound on its database. Running them as separate services isolates the LLM latency and failure profile from the Rust API's request path and allows independent scaling.

## Services in the sidecar

The Python sidecar provides six services:

| File | What it does |
|------|--------------|
| `clue_generator.py` | Generates clue text from a theme and location description |
| `difficulty.py` | Estimates how hard a clue will be for players |
| `semantic_match.py` | Semantic answer validation |
| `adaptive_hint.py` | Generates personalized hints based on session context |
| `session_analyzer.py` | Reviews completed sessions for insights |
| `photo_verifier.py` | Verifies photo answers against their description (vision-capable model) |

All six services call the model through one place: `app/llm.py`, a thin wrapper around [litellm](https://github.com/BerriAI/litellm). The provider is selected by the `LLM_MODEL` prefix in configuration (`anthropic/claude-opus-4-7`, `openai/gpt-4o`, `ollama/llava`, ...), so the sidecar is provider-agnostic — point it at a paid API, or at a free local model through Ollama. One caveat: model IDs date quickly — `claude-opus-4-7` is the shipped default at the time of writing and Anthropic already lists it among its legacy models, so treat `LLM_MODEL` as a per-release configuration decision rather than copying the ID from this page.

The code samples in this chapter are trimmed for readability — imports, error handling, and configuration plumbing are elided — but they follow the shapes of the real files in `backend-python/`.

```python
"""
Thin litellm wrapper — provider is selected by the LLM_MODEL prefix, e.g.:
  anthropic/claude-opus-4-7   → Anthropic (needs LLM_API_KEY)
  openai/gpt-4o               → OpenAI    (needs LLM_API_KEY)
  ollama/llama3               → Ollama local, text only (free, needs OLLAMA_BASE_URL)
  ollama/llava                → Ollama local, vision    (free, needs OLLAMA_BASE_URL)
"""
import litellm
from app.config import settings

litellm.drop_params = True  # ignore unsupported params per provider silently


async def complete(system: str, user: str, max_tokens: int = 512) -> str:
    resp = await litellm.acompletion(
        **_kwargs(max_tokens),
        messages=[
            {"role": "system", "content": system},
            {"role": "user",   "content": user},
        ],
    )
    return resp.choices[0].message.content.strip()
```

`_kwargs` assembles the model, token limit, and — when set — the API key and Ollama base URL from `app/config.py`. A separate `complete_vision` adds a base64 image to the message for photo verification.

## Clue generation

```python
_SYSTEM = """\
You are a treasure hunt designer. Given a location description and the correct answer,
write an engaging clue and a set of progressive hints.

Rules:
- The clue must not contain the answer.
- Hints must progress from vague to specific.
- Match the requested difficulty: 1 = trivial, 5 = very hard.
- Reply ONLY with valid JSON matching the schema:
  {"title": "...", "body": "...", "hints": ["...", "...", "..."]}
"""


async def generate(req: GenerateClueRequest) -> GenerateClueResponse:
    prompt = (
        f"Location/theme: {req.description}\n"
        f"Correct answer: {req.answer}\n"
        f"Difficulty: {req.difficulty}/5\n"
        f"Number of hints: {req.num_hints}"
    )
    text = await complete(_SYSTEM, prompt, max_tokens=512)
    return GenerateClueResponse(**json.loads(text))
```

The request and response are pydantic models (`GenerateClueRequest`, `GenerateClueResponse`), so the endpoint validates both directions.

The generator returns a structured response that the creator dashboard can pre-fill into the clue editor form. In the current UI, AI Draft fills the clue title, body, and hints. It does not overwrite the answer value or tolerance; the creator remains responsible for those validation fields before saving.

The critical HITL property: the AI cannot directly write to the clues table. Its output flows through the creator's review step. The human is a mandatory gate, not an optional one.

In practice, creators will approve AI-generated clues with varying levels of scrutiny. Some will edit heavily. Some will click "use as-is." The system cannot enforce careful review — it can only require that a human click a button. Whether that button-click constitutes meaningful review is outside the system's control.

## Difficulty estimation

```python
_SYSTEM = """\
You rate treasure hunt clue difficulty on a scale of 1 to 5.
1 = obvious answer, 5 = extremely cryptic.
Reply ONLY with valid JSON: {"score": <int 1-5>, "reasoning": "<one sentence>"}
"""


async def estimate(clue_body: str) -> DifficultyResponse:
    text = await complete(_SYSTEM, clue_body, max_tokens=128)
    data = json.loads(text)
    return DifficultyResponse(score=int(data["score"]), reasoning=data["reasoning"])
```

The current endpoint is a direct LLM judgment over the clue body. It does not receive the answer value, so it cannot compare clue-answer semantic distance. It is a useful prior — better than no estimate at all — but it may not match the population of players who will run the hunt.

The calibration loop (comparing estimates to actual session data) is described but not implemented.

## Semantic answer matching

The text validator in Rust uses Jaro-Winkler string similarity — a surface-level metric that works well for typos and abbreviations but fails for synonyms or paraphrases.

The Python sidecar provides a lightweight fallback that combines two signals: Jaccard token overlap (what fraction of words are shared?) and SequenceMatcher text similarity (how similar are the strings overall?), weighted 60/40:

```python
from difflib import SequenceMatcher
import re

def score(expected: str, submitted: str) -> float:
    """Lightweight semantic-ish similarity for local runtime. Returns 0–1."""
    left = expected.strip().lower()
    right = submitted.strip().lower()
    if not left or not right:
        return 0.0

    left_tokens = set(re.findall(r"\w+", left))
    right_tokens = set(re.findall(r"\w+", right))
    if not left_tokens or not right_tokens:
        token_overlap = 0.0
    else:
        token_overlap = len(left_tokens & right_tokens) / len(left_tokens | right_tokens)

    text_similarity = SequenceMatcher(None, left, right).ratio()
    return round((0.6 * token_overlap) + (0.4 * text_similarity), 4)
```

This is intentionally not an embedding-based model — it runs locally with zero dependencies, no GPU, and no API calls. "Large mountain in Asia" and "Everest" share no tokens (Jaccard = 0) and have low string similarity, so they would still fail here. The heuristic catches cases like "Mount Everest" vs "mount everest (Nepal)" where the core tokens overlap.

:::{admonition} Upgrade path: embeddings
:class: tip
For real semantic understanding — where "large mountain in Asia" should match "Everest" — replace the heuristic with an embedding model. A few lines of `sentence-transformers` would do it: encode both strings, compute cosine similarity, compare against the clue's tolerance. The tradeoff is adding a ~100 MB model download and ~10 ms of inference latency per check. The `/semantic-match` endpoint's contract (accept two strings, return a score) stays the same — only the implementation changes.
:::

## Photo verification

```python
_SYSTEM = """\
You are verifying whether a player photo matches a treasure hunt location description.
Respond ONLY with JSON: {"confidence": <float 0.0-1.0>, "reasoning": "<one sentence>"}
Be strict: only return confidence > 0.8 if the photo clearly shows the described location.
"""


async def verify(description: str, photo_b64: str) -> float:
    """Returns confidence 0–1 that the photo matches the description.

    Requires a vision-capable model — use ollama/llava for a free local option.
    """
    if "," in photo_b64:
        photo_b64 = photo_b64.split(",", 1)[1]
    text = await complete_vision(_SYSTEM, f"Location description: {description}", photo_b64)
    data = json.loads(text)
    return float(data["confidence"])
```

The photo verifier sends the image to a vision-capable model through the litellm wrapper, along with a natural-language description of what the photo should show. The response is reduced to a confidence score, and the Rust validator compares that score with the clue's `answer_tolerance`.

This is fragile in production. A production photo verifier would need stronger structured output, calibrated confidence scores, and a fallback path when the model is uncertain. In this implementation, the result is acted on synchronously: if the confidence is below tolerance, the photo answer is rejected.

## The AiClient in Rust

The Rust API communicates with the Python sidecar through `services/ai_client.rs`:

```rust
pub struct AiClient {
    base_url: String,
    client:   reqwest::Client,
}

impl AiClient {
    pub async fn generate_clue(&self, req: GenerateClueRequest) -> Result<GeneratedClue> {
        let resp = self.client
            .post(format!("{}/generate-clue", self.base_url))
            .json(&req)
            .send().await?
            .json::<GeneratedClue>().await?;
        Ok(resp)
    }

    pub async fn estimate_difficulty(&self, req: DifficultyRequest) -> Result<DifficultyEstimate> {
        // ...
    }
}
```

The client is injected into `AppState` and available to handlers and validators. The current methods are:

- `semantic_match` → Python `/semantic-match`, used as text-answer fallback
- `verify_photo` → Python `/verify-photo`, used synchronously for photo clues
- `generate_clue` → Python `/generate-clue`, exposed through `POST /hunts/:id/ai/generate-clue`
- `estimate_difficulty` → Python `/estimate-difficulty`, exposed through `POST /hunts/:id/ai/difficulty`
- `adaptive_hint` → Python `/adaptive-hint`, used after static hints are exhausted
- `analyze_session` → Python `/analyze-session`, exposed through `GET /sessions/:id/analysis`

## Reflection

Every call to the AI sidecar has a latency cost, a financial cost (API tokens — with a paid provider someone pays for every call, and in a classroom deployment that someone is typically the instructor or institution; a local Ollama model removes the per-call bill but not the latency or the hardware requirement), and a reliability risk (the sidecar might be down or slow). The current design makes these calls synchronously — the request waits for the AI response.

For photo verification, this is probably fine. For adaptive hints, it might add noticeable latency to a step players are already frustrated by.

How would you architect the AI calls differently? What would an asynchronous, queue-based approach look like? What would it make better, and what would it make worse? (And a harder question: for which tasks would removing the AI entirely make the system *better*? Not every loop benefits from an LLM — sometimes a well-tuned heuristic outperforms a language model at lower cost and with fewer failure modes.)
